Q: Why two tables instead of one?
A: Separation of concerns:
   - bill_sponsors: Volatile (sponsors change per bill, per session)
   - wy_legislators: Relatively stable (one rep per seat, per session)
   - Normalized structure enables queries like "What bills did senator X sponsor?"

Q: Why include chamber in bill_sponsors if we can join wy_legislators?
A: Denormalization for speed. Most queries will return <10 sponsors. Joining
   would add complexity. One-way denormalization is acceptable trade-off.

Q: Why start with county-based delegation instead of full geocoding?
A: MVP (minimum viable product) approach:
   - Phase 2: Counties only (manual mapping, ~23 WY counties)
   - Phase 2b: Auto-geocoding if needed (address â†’ lat/long â†’ district)
   - Reduces Phase 2 scope; easier to extend later

Q: Why JSON for county_assignment instead of separate table?
A: Flexibility:
   - One rep might represent 1, 2, or 3 counties
   - JSON array handles all cases cleanly
   - Avoids need for intermediary table (representative_counties)
   - Can be NULL if not yet mapped (Phase 2a)

Q: Why include legislative_session in wy_legislators?
A: Multi-year tracking:
   - Wyoming legislature meets every 2 years
   - 2025 â†’ 2027 â†’ 2029, etc.
   - Want historical data (who was my rep in 2023?)
   - Composite key (seat_id, session) enables time-series queries
